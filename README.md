## 1. Product & Scope (one-page summary)

**Breeez** is a South African single-brand ecommerce store that sells across 10 categories (Apparel, Bags, Bathroom, Bedroom, Curtains, Everyday Essentials, Home Decor, Kitchen, Shoes, Back-to-School). It runs as a web app at a Vercel-hosted URL, as an Apple App Store app, and as a Google Play Store app — all from **one Next.js codebase wrapped with Capacitor**.

**Reference flow:** Trends Day to Day (trendsdaytoday.co.za) — verified by scraping their home + PLP + PDP. **Industry-standard ecommerce layout patterns** (category tile grid, PLP with sidebar filters + grid + sort, PDP with variants + "Select options", slide-out cart drawer, account drawer, wishlist, compare) — these are universal patterns used by every modern ecommerce site and are not copyrightable. **Original Breeez visual design** — own typography, colour palette, copy voice, photography style. Explicitly NOT a clone of Trends' UI.

**Out of scope for v1:** seller/marketplace features, multi-currency, multi-language, loyalty programme, subscriptions, B2B/wholesale.

## 2. Personas

| Persona | Who | What they do |
|---|---|---|
| **Shopper** | Anonymous SA customer | Browse, search, add to cart, checkout, track order, manage wishlist + addresses |
| **Customer** | Registered SA customer (same as shopper, but logged in) | All shopper actions + order history, saved addresses, password reset, returns initiation |
| **Admin (Chris)** | You — single admin account | Add/edit products + variants, manage categories, view orders, mark orders shipped, process returns, view sales dashboard |
| **Support (you, on phone)** | You in support mode | Look up an order by order # or email, see status, refund, edit address pre-shipment |

**No public seller signup.** No multi-tenancy. Single-admin model for v1.

## 3. Data model (Postgres via Supabase)

All money is stored as integer cents (ZAR × 100). All timestamps are `timestamptz` UTC.

### 3.1 Reference tables

```
categories
  id          uuid pk
  slug        text unique           -- 'apparel', 'bedroom', etc.
  name        text                  -- 'Apparel'
  description text nullable
  image_url   text nullable
  parent_id   uuid fk categories    -- nullable, for sub-categories (Women/Men/Kids under Apparel)
  sort_order  int default 0
  is_active   bool default true
  created_at  timestamptz
  updated_at  timestamptz

products
  id              uuid pk
  slug            text unique           -- 'girls-raspberry-jogger'
  name            text
  description     text                  -- markdown
  category_id     uuid fk categories
  brand           text default 'Breeez'
  base_price_cents int                   -- ZAR × 100
  compare_at_cents int nullable         -- for "was R169, now R129"

  updated_at      timestamptz

  id           uuid pk
```

### 3.5 Row-Level Security (RLS)

- **products, categories, product_variants, product_images**: `select` allowed to `anon` and `authenticated`; write requires `auth.uid() IN (SELECT id FROM admins)`.
- **customers, addresses, wishlists, carts, cart_items**: customer can only see/edit their own row (`auth.uid() = customer_id`).
- **orders, order_items, order_events, returns**: customer sees their own; admin sees all.
- **discount_codes**: read by anyone (for validation); write by admin only.

The `admins` table is a simple allowlist:

```
admins
  user_id  uuid pk fk auth.users
  created_at timestamptz
  -- You add yourself here manually after first sign-up.
```

## 5. Auth model

**Decision:** Supabase Auth with **email + password** as primary, **magic link** as fallback.

- Customer signs up with email + password (min 8 chars, 1 number)
- Supabase sends verification email (Resend SMTP)
- Login by email + password OR "email me a magic link"
- Password reset via magic link
- Session = Supabase JWT in httpOnly cookie (web) / SecureStorage (native)
- **Admin**: same Supabase Auth, but `user_id` must exist in `admins` table to access `/admin/*`

**Why not social login (Google/Apple/Facebook)?** Add it later if needed. v1 is email-only to keep scope tight and POPIA-clear.

## 6. Payments

**Primary:** PayFast — supports card, Instant EFT, SnapScan, Zapper, normal EFT, and mobile money from one integration. Most popular SA gateway for SMBs.

**Secondary:** Yoco (online card payments) — used as fallback if PayFast declines a card.

**Tertiary:** Ozow — Instant EFT for customers who hate card forms.

All three are integrated as **Supabase Edge Functions** that:
1. Create a payment intent on the gateway
2. Redirect customer to gateway-hosted page (PayFast/Ozow) or in-page modal (Yoco)
3. Receive webhook callback → update `orders.status = 'paid'`, send confirmation email, decrement stock

**No card data ever touches Breeez servers** — gateways handle PCI compliance. Supabase Edge Functions only handle redirect URLs + webhook verification.

**Money flow:** PayFast/Yoco/Ozow settle into your bank account on their schedule (PayFast = T+2, Yoco = T+1, Ozow = T+1).

## 7. Shipping

**Three options at checkout** (customer picks one):

| Option | Provider | Cost | When |
|---|---|---|---|
| **Pargo pickup point** | Pargo | R49 | Always available. Customer picks nearest of 2,500+ points. |
| **Door delivery (standard)** | The Courier Guy | R89 | 2-5 working days nationwide |
| **Same-day metro** | Dawn Wing | R149 | Major metros only (JHB, CPT, DBN, PTA). Order before 11am. |

**Free delivery** when subtotal ≥ R500 (configurable constant).

**Address capture:** customer enters address inline at checkout OR picks from address book if logged in.

**Courier integration:**
- **Pargo**: their API gives us a list of nearby pickup points by postal code
- **The Courier Guy + Dawn Wing**: address-based booking; we POST a booking and get a waybill #, then poll/webhook for status updates

**Tracking:** stored on `orders.shipping_tracking`. Customer sees it on order detail page. Status updates push via email + (if native) push notification.

## 8. Search & discovery

**v1:** Postgres full-text search on `products.name + description + tags`, with simple category + price filters.

**Migration path:** swap in Meilisearch when:
- Catalogue > 5,000 SKUs, OR
- We want typo-tolerant search ("rasbery" → "raspberry"), OR
- We want faceted search with hit counts per attribute

Meilisearch has a generous free tier and runs on Fly.io / Render free tier. Not a v1 concern.

## 9. Images

**Storage:** Supabase Storage, public bucket `product-images`.

**Upload:** admin uploads via admin UI → stored at `product-images/{product_id}/{filename}`.

**Transforms:** Supabase Storage's built-in image transformation API. We render responsive sizes:
- `?width=80` — cart thumbnail
- `?width=300` — product card (PLP)
- `?width=800` — PDP hero

Lazy-loaded with native `<img loading="lazy">`.

**Format:** WebP with JPEG fallback (Supabase Storage's transform handles automatically).

## 12. Analytics

**Tool:** Plausible (self-hosted, R50/mo on Hetzner) OR PostHog cloud (free tier: 1M events/mo).

**Decision:** PostHog cloud for v1. Zero infra, covers pageviews + custom events + funnels.

**Events tracked:**
- `product_viewed`
- `variant_selected`
- `added_to_cart`
- `checkout_started`
- `payment_initiated`
- `purchase_completed` (with order value)
- `searched`
- `wishlist_added`

No PII in events. Cookie-less.

## 13. Admin

**v1:** Use **Supabase Studio** (the built-in dashboard at `https://app.supabase.com`) for direct data access. Build **only the custom screens** in `/admin/*` for daily workflows (orders, products, returns).

This means:
- For ad-hoc data fixes, you go to Supabase Studio (table editor)
- For daily operations (fulfilling orders, adding stock, processing returns), you use `/admin/*` in your own Breeez admin

**Why not Supabase Studio for everything?** Because it's ugly, slow for bulk operations, and you can't ship a "View your dashboard" link to a future ops hire without giving them Supabase access.

**v2 (later):** Build a full custom admin if Supabase Studio becomes a bottleneck. Not a v1 concern.

## 16. POPIA (Protection of Personal Information Act) compliance

You're collecting personal info from SA customers. POPIA requires:

- [x] **Consent** at signup (you agree to our privacy policy + T&Cs)
- [x] **Purpose limitation** — only collect what you need for the order (name, email, phone, address)
- [x] **Minimal data** — don't ask for ID, birthdate, etc.
- [x] **Secure storage** — Supabase encrypts at rest + in transit (TLS)
- [x] **Access control** — RLS prevents customer A seeing customer B's data
- [x] **Data retention policy** — delete accounts that haven't ordered in 3 years (cron job)
- [x] **Breach notification process** — if Supabase is breached, notify customers within 72h
- [x] **Right to be forgotten** — `/account/delete` endpoint deletes account + orders after 30-day grace
- [x] **POPI Act registration** — Info Regulator (you do this once as a business, not per app)
- [x] **Privacy notice** at `/legal/privacy` listing exactly what you collect + why
